# ADR 0001: Решение об оптимизации задержки inventory forecast

**Дата:** 2026-05-24  
**Статус:** Принято  
**Автор:** MLOps HW9

---

## Контекст

ML-система расчёта складских запасов предоставляет прогнозы через API.  
В ходе мониторинга зафиксирован рост p95-задержки с ~264 мс до ~1376 мс  
(≈5.2× деградация). Задержка превысила нормальный SLO (300 мс) и приблизилась  
к критическому порогу (1000 мс), что угрожает соблюдению SLA для торговых точек.

Метрика выбрана как **системная** (p95 end-to-end latency), а не только  
модельная (MAPE/accuracy), что соответствует принципам Metrics Driven Development:  
решение принимается на уровне всей системы в контексте бизнеса.

## Статистическое обоснование

| Параметр | Значение |
|---|---|
| Тест | Манн–Уитни U (двусторонний) |
| N референс | 200 наблюдений |
| N новые данные | 200 наблюдений |
| p95 референс | 263.54 мс |
| p95 новые данные | 1376.45 мс |
| Δp95 | +1112.91 мс |
| p-value | < 0.001 |
| α | 0.05 |
| Статистически значимо | Да |
| Превышен SLO (300 мс) | Да |

**Вывод:** нулевая гипотеза (распределения одинаковы) отвергается.  
Деградация статистически значима и практически значима (SLO нарушен).

Подробности: [`reports/mdd_test_result.md`](../reports/mdd_test_result.md)  
График: [`reports/mdd_latency_distribution.png`](../reports/mdd_latency_distribution.png)

## Решение

Принять факт деградации и реализовать следующие архитектурные улучшения:

1. **Кеш предсказаний** (Redis / in-memory TTL 5 мин) перед обращением к модели  
   для часто запрашиваемых SKU.

2. **Предрасчёт лаговых признаков** в батч-задаче DAG  
   вместо вычисления их онлайн в момент запроса.

3. **Удержание предыдущей стабильной модели** в production  
   до устранения узкого места (версия v1 остаётся активной).

4. **Усиленный мониторинг** p95-latency с алертом при превышении 300 мс  
   более чем на 3 минуты подряд.

## Последствия

**Положительные:**
- Ожидаемое снижение p95 ниже 300 мс после внедрения кеша.
- Снижение нагрузки на модель (кеш поглощает повторные запросы).
- Прозрачная трассируемость решения через ADR + статистический отчёт.

**Отрицательные / риски:**
- Введение кеша добавляет компонент инфраструктуры (Redis).
- Устаревшие кешированные значения ≤5 мин при резком изменении остатков.
- Потребуется обновление мониторинга (новый SLI для hit-rate кеша).
